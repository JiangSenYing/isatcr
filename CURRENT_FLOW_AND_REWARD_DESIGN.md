# 当前流程与奖励设计

> 整理范围：当前主入口 `PRC_GNN_TRANSFORMER.py`、默认配置 `train/test_transformer.yaml`，以及工作区现有的 LEO–MEO 域入口奖励实现。本文描述的是**当前代码实际行为**，不是早期实验配置或仅存在于注释中的设想。

## 1. 系统概览

当前系统采用 LEO 与 MEO 两层联合决策：

- **LEO Agent**：以单颗 LEO 为决策节点，选择邻居转发或本地计算，负责域内逐跳路由和计算卸载。
- **MEO Agent**：以 MEO 域为决策粒度，选择下一 MEO 域及相应跨域入口，给 LEO 设置阶段性目标。
- **全局 Transformer**：配置中 `enabled: false`，因此当前不训练也不使用未来网络状态预测；但 `meo_exit_enabled: true`，所以 MEO 域级 Agent 仍会创建并参与决策。
- **仿真环境**：基于 SimPy 推进传输、排队、计算、链路变化和下行事件；外层每次 `env.step()` 推进 1 秒。

当前默认实验的关键设置如下：

| 项目 | 当前值 |
|---|---:|
| 运行阶段 | `train` |
| LEO 算法 | GNN-DQN / DDQN 更新逻辑 |
| LEO 观测 | `graph`，2-hop |
| LEO 动作数 | 5 |
| MEO Agent | 启用，自注意力编码器 |
| 全局 Transformer | 关闭 |
| LEO 折扣因子 | 0.99 |
| MEO 折扣因子 | 0.97 |
| 仿真步长 | 1 秒 |
| 每步 LEO 参数更新次数 | 5 |
| 每次 MEO 更新次数 | 1 |

## 2. 端到端运行流程

```text
生成业务包
  │
  ├─ 源域 = 目的域：直接进入 LEO 逐跳决策
  │
  └─ 源域 ≠ 目的域：MEO Agent 选择下一域和跨域入口
          │
          ▼
LEO Agent 在当前节点决策
  ├─ 动作 0～3：转发给对应邻居
  ├─ 动作 4：在本节点计算，再回到转发队列
  └─ 非法/不可执行：终止并给予丢包奖励
          │
          ▼
到达 MEO 指定域入口？
  ├─ 否：继续 LEO 逐跳决策
  └─ 是：结算 LEO 域入口奖励和 MEO 分段奖励
           ├─ 尚未进入目的域：MEO 再规划下一段
           └─ 已进入目的域：取消临时入口，继续前往最终下传卫星
          │
          ▼
到达最终卫星并完成星地下行
  ├─ 已计算：正常到达奖励
  ├─ 未计算：异常到达奖励（无基础成功奖）
  └─ 中途失败：丢包奖励
          │
          ▼
经验进入 LEO/MEO 各自回放池并执行网络更新
```

### 2.1 业务包生成与首次跨域规划

1. 环境按业务到达过程选择源 LEO、任务类型、数据量、计算需求、计算后数据量和最终目的 LEO。
2. 数据包初始状态为 `is_computed = false`，并令 `leo_domain_entry_time = creation_time`。
3. 当源、目的 LEO 隶属不同 MEO 域时，源域 MEO Agent 选择下一域，并返回一个或多个目标边界 LEO，写入 `packet.temporary_destination`。
4. MEO 决策的状态、动作、动作掩码、域距离、边界代价和任务上下文被附着到数据包，等待分段到达或失败后生成经验。

### 2.2 LEO 单步决策

每次数据包从 LEO 的 `forward_queue` 取出后：

1. 跳数加 1，并等待节点处理时间。
2. 根据当前节点、邻居及链路状态、任务属性、跳数、计算状态和当前目标构造图观测。
3. 若存在 MEO 临时入口，LEO 的当前目标是入口候选；否则目标是最终下传卫星。
4. Agent 对候选目标及动作打分，执行 epsilon-greedy 选择。

动作含义：

| 动作 | 含义 | 后续处理 |
|---|---|---|
| 0～3 | 转发至对应邻居 | 进入发送队列，经历排队、发送和传播 |
| 4 | 本地计算 | 本次不计转发跳，进入计算队列，完成后回到转发队列 |
| 其他/不可执行 | 非法动作 | 数据包终止，记录终止经验和丢包奖励 |

已完成计算时，策略只允许使用转发动作；未完成计算时可以选择本地计算。当前配置没有显式设置 `leo_action_mask_enabled`，因而采用代码默认值 `false`；但动作选择代码仍包含针对 `is_computed` 的合法动作约束分支。

### 2.3 计算流程

选择动作 4 后，计算耗时为：

```text
计算耗时 = 本次计算需求 / 节点计算能力
```

计算结束后：

- 更新计算等待时间和资源占用；
- 数据包大小变为 `size_after_computing`；
- 当前实现将包内 `is_computed` 写为 `true`；
- 数据包重新进入当前节点的转发队列。

计算动作本身没有额外的固定正奖励，其收益通过数据缩小、后续时延变化，以及域入口/最终到达时“已计算”与“未计算”的奖励差异体现。

### 2.4 跨域入口与连续子任务

跨域传输成功、数据包被新域边界 LEO 接收时，环境记录：

```text
leo_previous_domain_entry_time = 上一域入口时间
leo_domain_entry_time          = 本次跨域接收时间
meo_segment_time               = 本次跨域接收时间
```

随后，新域边界节点识别自己为 `temporary_destination`，完成以下操作：

1. 按上一域的实际耗时结算 LEO 域入口奖励；
2. 为上一条 LEO 动作写入入口经验；
3. 将该入口奖励传给 MEO Agent，结算 MEO 分段经验；
4. 若尚未进入目的域，MEO 再选择下一域与入口；
5. 若已进入目的域，不再设置跨域入口，LEO 继续完成“目的域入口 → 最终下传卫星”的最后一段。

普通域入口对 LEO 来说是阶段边界而不是 episode 终点，使用 `done = 0`；只有入口节点同时等于最终目的 LEO 时才使用 `done = 1`。

### 2.5 最终下传与失败

- 数据包到达最终 LEO 后进入下行队列，经历发送耗时和固定下行传播时延，再结算最终奖励。
- 路由队列/内存溢出、下一跳或链路失效、节点失效、非法动作、跳数超限、星间传输中断和下传中断均按终止失败处理。
- LEO 的入口、最终和失败奖励均使用**当前 LEO 域内时延**；端到端时延仍单独用于业务统计及 MEO 终局时延项。

## 3. LEO 奖励设计

当前配置：

```yaml
reward_factors: [1, 0.05, 1, 0.1, 0.25]
# [reach_factor, delay_factor, loss_factor,
#  memory_threshold, memory_factor]
```

记：

- \(\Delta t_{step}\)：上一动作到当前决策的单步耗时；
- \(\Delta t_{domain}\)：当前或刚完成的 LEO 域段耗时；
- \(m\)：当前节点剩余内存比例；
- \(\mathbb{1}[\cdot]\)：条件成立时为 1，否则为 0。

### 3.1 普通逐步奖励

用于成功执行的中间转发动作和本地计算动作：

$$
R_{normal}=-0.05\Delta t_{step}-0.25\mathbb{1}[m<0.1]
$$

含义：持续惩罚耗时；当前节点剩余内存低于 10% 时，再扣 0.25。该公式不直接区分数据包是否已经计算。

### 3.2 域入口奖励

到达 MEO 指定的下一域入口时：

$$
R_{entry}=\begin{cases}
1-0.05\Delta t_{domain}, & \text{已计算}\\
-0.05\Delta t_{domain}, & \text{未计算}
\end{cases}
$$

已计算与未计算之间固定相差 1。普通入口经验 `done = 0`，因此 DDQN 可以从新域状态继续 bootstrap；入口动作结算后会清除上一动作引用，避免同一动作又获得一次普通逐步奖励。

### 3.3 最终下传奖励

最终下传使用与入口相同的两种到达公式，但时延是“进入当前目的域后到完成下行”的耗时：

$$
R_{final}=\begin{cases}
1-0.05\Delta t_{domain}, & \text{已计算并到达}\\
-0.05\Delta t_{domain}, & \text{未计算但到达}
\end{cases}
$$

最终下传经验使用 `done = 1`。未完成计算的数据包即使成功到达，也没有基础成功奖励。

### 3.4 失败奖励

所有可归因于当前路由/计算过程的终止失败统一为：

$$
R_{loss}=-1-0.05\Delta t_{domain}
$$

失败经验使用 `done = 1`。非法动作由当前非法动作承担终止惩罚；到达当前节点的上一动作先按普通奖励结算，避免错误归因。

### 3.5 LEO 奖励归因总结

| 事件 | reward | done | 时延口径 |
|---|---:|---:|---|
| 普通转发/计算后的下一决策 | `-0.05Δt - 内存惩罚` | 0 | 单步 |
| 已计算后到达普通域入口 | `1 - 0.05Δt` | 0 | 上一域段 |
| 未计算时到达普通域入口 | `-0.05Δt` | 0 | 上一域段 |
| 已计算后最终下传 | `1 - 0.05Δt` | 1 | 当前目的域段 |
| 未计算但最终下传 | `-0.05Δt` | 1 | 当前目的域段 |
| 中途失败 | `-1 - 0.05Δt` | 1 | 当前域段 |

## 4. MEO 奖励设计

MEO Agent 使用独立奖励，不直接把 LEO reward 当成自己的完整训练目标。它有两种结算时机：成功到达下一域入口时的**分段奖励**，以及在仍有活动 MEO 决策时发生包级终止的**终局奖励**。

### 4.1 通用默认公式

MEO 分段奖励：

$$
R_{MEO}^{seg}=r_{seg}
+w_T R_{LEO}^{entry}
+w_P(\gamma\phi(s')-\phi(s))
-w_DH_{domain}
-w_PH_{path}
-w_LL_{boundary}
-w_{delay}\Delta t_{seg}
$$

其中 \(\phi(s)=-d(s, target)\)，即以到目标域的最短域距离构造势函数。

MEO 终局奖励：

$$
R_{MEO}^{terminal}=R_{outcome}+w_T R_{LEO}^{terminal}
-w_S score-w_DH_{domain}-w_LL_{boundary}-w_{delay}\Delta t_{e2e}
$$

### 4.2 当前配置下的实际分段公式

当前非零参数只有：

```text
terminal_reward_weight = 0.2
progress_reward_weight = 0.02
boundary_delay_penalty = 0.08
gamma = 0.97
```

`segment_success_reward`、域跳惩罚、路径跳数惩罚和边界负载惩罚均为 0，因此实际为：

$$
R_{MEO}^{seg}=0.2R_{LEO}^{entry}
+0.02\left(0.97\phi(s')-\phi(s)\right)
-0.08\Delta t_{seg}
$$

这意味着当前 MEO 分段奖励主要鼓励：

- 让 LEO 以更高入口奖励到达下一域，间接偏好已完成计算且耗时较短的方案；
- 缩短到目标域的剩余域距离；
- 降低本段实际耗时。

### 4.3 当前配置下的实际终局公式

终局结果基础项为：

| 结果 | \(R_{outcome}\) |
|---|---:|
| 已计算并到达 | 1.5 |
| 未计算但到达 | `1.5 - 1.5 = 0` |
| 内存溢出丢包 | `-1.5 - 0.5 = -2.0` |
| 链路/普通失败 | -1.5 |

当前 `score_penalty`、域跳惩罚和边界负载惩罚均为 0，所以：

$$
R_{MEO}^{terminal}=R_{outcome}+0.2R_{LEO}^{terminal}-0.08\Delta t_{e2e}
$$

需要注意当前结算语义：到达每个计划入口时，对应 MEO trace 会立即通过 `finish_segment()` 结算并移出活动列表。进入目的 MEO 域时若不再产生下一段计划，该分段经验直接 `done = 1`。之后最终 LEO 下传通常已没有活动 MEO trace，因此不会再新建一条 MEO 终局经验；终局公式主要用于包在活动 MEO 分段内提前成功/失败终止的情况。`terminal_credit_weight = 0`，所以当前也不会把最终结果追加回已经完成的 MEO 分段经验。

## 5. 经验组织与训练

### 5.1 LEO 经验

当前 `obs_type: graph`，经验格式为：

```text
[obs, is_computed, action, reward, next_obs, done]
```

`is_computed` 作为 mark 一并存储。每个外层 step：

1. 环境收集这一秒内产生的 LEO 经验；
2. 主循环将经验写入 LEO 回放池；
3. LEO Agent 连续执行 5 次参数更新；
4. epsilon 从 0.9 按每步 0.98 衰减，最低为 0.02；
5. 目标网络每 100 次更新同步一次。

### 5.2 MEO 经验

MEO 经验包含域级状态、被选下一域动作、reward、下一域状态、done 和下一状态动作掩码。当前：

- 每产生新的 MEO 决策即可尝试更新；
- 每次执行 1 次更新；
- epsilon 从 0.9 衰减到最低 0.01；
- 只有成功完成梯度更新后才衰减 epsilon；
- 目标网络每 10 次梯度更新同步一次。

### 5.3 信用传播

- LEO 普通域入口使用 `done = 0`，可以沿整个跨域业务持续传播最终价值。
- MEO 每段单独形成经验；若还有下一段，则以新 MEO 决策状态作为 `next_state`。
- 当前 MEO 的 `terminal_credit_weight = 0`，已完成分段不会获得额外包级终局信用。

## 6. 时延口径

当前实现同时维护三种时延，不能混用：

| 时延 | 定义 | 用途 |
|---|---|---|
| 单步时延 | 当前决策时间减上一动作记录时间 | LEO 普通逐步奖励 |
| LEO 域段时延 | 当前域入口/终点减本域入口时间 | LEO 域入口、最终及失败奖励 |
| 端到端时延 | 包终止时间减创建时间 | MEO 终局时延惩罚、业务统计 |

跨域时间戳只在数据包成功进入不同 MEO 域的下一跳队列后更新；同域转发、入队失败或传播失败不会错误开启新域计时。

## 7. 日志与评价指标

当前日志将不同奖励分开统计：

- `Average ending reward`：只统计最终下传和终止失败的 LEO reward；
- `Average domain-entry reward`：统计非终止跨域入口 reward，以及尚未进入目的域时发生的阶段失败；
- `MEO Agent ... reward_avg`：统计 MEO Agent 实际写入回放池的分段/终局 reward；
- `round_reward`：主训练循环中收到的全部 LEO 经验 reward 之和，包含普通逐步、入口和终局经验。

同时记录包成功率、丢包率、端到端平均时延、平均跳数、计算比例、计算等待时间、内存溢出和非法动作数量。

## 9. 一句话总结

当前设计用 MEO Agent 把长距离跨域业务拆成连续的“到达下一域入口”子任务，再由 LEO Agent 联合完成逐跳路由与本地计算；LEO 奖励以域内耗时、计算完成状态和失败为核心，MEO 奖励则融合 LEO 入口结果、域距离进展和实际分段时延。
